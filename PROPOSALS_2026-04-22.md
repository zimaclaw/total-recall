## 🎯 Проблема

1. **Сессии "пропадают"** — total-recall фильтрует агенты по `agentId`, записывает только `main`
2. **KB пуст** — 1 тестовая запись, нет автоматического наполнения
3. **Нет критериев** что важно сохранять, что нет
4. **Риск потери данных** — пользователь может вернуться к сессии, а она уже архивирована

## 🏗️ Предлагаемая архитектура: Гибридная Архивация

### Концепция

**Два механизма для разных сценариев:**

1. **Ручной handoff** — для текущих активных сессий (пользователь контролирует)
2. **Cron-based архивация** — для неактивных сессий (автоматический fallback)

**Ключевое правило:** Handoff блокирует cron для этой сессии — нет дублирования.

---

### Поток данных

```
┌─────────────────────────────────────────────────────────────┐
│                  ТЕКУЩАЯ СЕССИЯ (активна)                   │
└─────────────────────────────────────────────────────────────┘
                            ↓
              Пользователь: /kb handoff [результат]
                            ↓
                    Создание KB записи:
                      ├── title: из handoff
                      ├── summary: handoff + контекст сессии
                      ├── content: последние N сообщений
                      ├── source_tool: "handoff"
                      └── category: inferred из контента
                            ↓
                    Создание mapping: sessionId → kb_id
                            ↓
              Опционально: /kb handoff --new-session
                            └── Создание новой сессии с чистым контекстом

┌─────────────────────────────────────────────────────────────┐
│              АРХИВНЫЕ СЕССИИ (неактивны > 48 часов)          │
└─────────────────────────────────────────────────────────────┘
                            ↓
                    Cron job (ежедневно в 3 утра)
                            ↓
                    SQL запрос:
                      SELECT sessionId FROM sessions
                      WHERE last_accessed_at < NOW() - INTERVAL '48 hours'
                      AND agentId IN (archiveAgents)
                      AND sessionId NOT IN (SELECT sessionId FROM kb_session_mapping)
                            ↓
                    Для каждой сессии:
                      ├── Быстрый фильтр (сообщения > N, ключевые слова)
                      ├── LLM анализ (последние 10K токенов)
                      ├── Проверка дубликатов через векторный поиск
                      └── Создание KB записи в kb_hot (14 дней)
                          ├── source_tool: "cron-archive"
                          └── category: inferred из контента
```

## 📊 Сравнение подходов

| Аспект | Только cron | Только handoff | Гибридный (принято) |
|--------|-------------|----------------|---------------------|
| **Контроль** | Нет (автоматически) | Полный | Частичный (текущие сессии) |
| **Автоматизация** | Полная | Нет | Частичная (архивные сессии) |
| **Дублирование** | Нет | Нет | Нет (handoff блокирует cron) |
| **Задержка** | До 48+ часов | Мгновенно | Мгновенно (handoff) + 48ч (cron) |
| **Потеря данных** | Низкая | Высокая (забыл handoff) | Низкая (cron как fallback) |

## 🔧 План реализации

### Фаза 1: Ручной handoff (P1)

1. **Команда `/kb handoff`** — 3 часа
   - Парсинг аргументов
   - Сбор контекста сессии
   - Создание KB записи
   - Шаблон handoff для пользователя

2. **Таблица `kb_session_mapping`** — 1 час
   - sessionId → kb_id
   - source_type (handoff/cron)
   - created_at

**Итого:** 4 часа

---

### Фаза 2: Cron-based архивация (P2)

1. **Поле `last_accessed_at` в sessions** — 1 час
   - Обновление при `before_prompt_build`
   - Fallback на `created_at` для старых сессий

2. **Cron job с проверкой mapping** — 2 часа
   - SQL запрос неактивных сессий
   - Исключение уже архивированных
   - Логирование

3. **Двухэтапная проверка** — 3 часа
   - Быстрый фильтр (сообщения, ключевые слова)
   - LLM анализ для кандидатов

4. **LLM извлечение контента** — 4 часа
   - Промпты для каждого типа контента
   - Валидация JSON
   - Fallback на простой текст

5. **Проверка дубликатов** — 1 час
   - Векторный поиск через Qdrant
   - Порог similarity > 0.85

**Итого:** 11 часов

---

### Фаза 3: Оптимизация (P3)

1. **Подсказка handoff для длинных сессий** — 2 часа
   - Inject в `prependSystemContext`
   - Один раз на сессию после 6 часов

2. **Статистика использования** — 2 часа
   - Таблица `kb_archive_stats`
   - Команда `/kb stats`

3. **Интеграция с Curator flashback** — 2 часа
   - KB как fallback после Neo4j
   - Ограничение токенов (2-4K)

**Итого:** 6 часов

## ⚠️ Риски и митигация

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Перегрузка KB малоценной информацией | Средняя | Высокое | Строгие критерии, двухэтапная проверка |
| Пользователь не доверяет автоматике | Средняя | Среднее | Прозрачное логирование, возможность отмены |
| Сложность отладки при множестве триггеров | Высокая | Среднее | Детальное логирование, тестовые сессии |
| LLM ошибки в классификации | Средняя | Низкое | Валидация JSON, fallback на general |
| Потеря данных при таймауте | Низкая | Высокое | Cron как fallback, резервное копирование |
