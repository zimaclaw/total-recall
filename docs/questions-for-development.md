### ✅ Результаты обсуждения (2026-03-26 12:07 UTC)

**Что определено:**

1. **Дерево решений понятно** — базовая структура работает
2. **Фильтр "нужно ли хранить"** — не все знания сохраняем:
   - Пожелания по конкретным случаям → не хранить
   - Временная информация → не хранить
   - Общие уроки, инструкции, конфигурации → хранить
3. **Категория "переданные знания"** — опыт от Олега (мозг → мозг):
   - Не урок из задачи, не инструкция, не правило
   - Важные знания которые Олег передаёт напрямую
   - Примеры: "Система должна быть надёжной", "Приватность прежде всего"

**Обновлённое дерево решений:**

```
Куда записать знание?

0. НУЖНО ЛИ ХРАНИТЬ ЭТО ЗНАНИЕ?
   → Это пожелание по конкретному случаю? → Нет (не хранить)
   → Это временная информация? → Нет (не хранить)
   → Да (хранить): иди к пункту 1

1. Это урок из опыта (после завершённой задачи)?
   → Да: Mem0 (Lesson/Principle/Conclusion)
   → Нет: иди к пункту 2

2. Это правило поведения (обязательно соблюдать)?
   → Да: RULES.md или SOUL.md
   → Нет: иди к пункту 3

3. Это "переданное знание" от Олега (опыт/обобщения мозг→мозг)?
   → Да: total-recall/docs/transferred/<topic>.md
   → Нет: иди к пункту 4

4. Это инструкция "как сделать что-то"?
   → Да: total-recall/docs/instructions/<topic>.md
   → Нет: иди к пункту 5

5. Это конфигурация системы?
   → Да: total-recall/docs/configurations/<system>.md
   → Нет: иди к пункту 6

6. Это решение конкретной проблемы?
   → Да: total-recall/docs/solutions/<problem>.md
   → Нет: иди к пункту 7

7. Это инсайт о Олеге или системе?
   → Да: MEMORY.md
   → Нет: не сохранять (или обсудить с Олегом)
```

**Что остаётся ОТКРЫТЫМ:**

- ❓ Как интегрировать с существующей системой — Mem0, RULES.md, MEMORY.md

## 2. Проблема: "Инструкции и конфигурации"

**Наблюдение:**
Существующая система хранит:
- **Mem0** — уроки из опыта (после задач с outcome)
- **RULES.md** — правила поведения
- **MEMORY.md** — инсайты и выводы
- **SYSTEM.md** — технический контекст

**Но нет места для:**
- Инструкций "как сделать" (например: "как настроить ollama с FLASH_ATTENTION")
- Конфигураций (например: docker-compose.yml для ollama)
- Решений проблем (например: "VRAM оптимизация qwen3:14b")
- Важных правил конфигурации (например: "контекст кратный 512")

**Вопрос для total-recall:**
Где хранить инструкции, конфигурации и решения проблем?

**Предложение:**
Добавить в total-recall/docs разделы:
- `instructions/` — инструкции "как сделать"
- `configurations/` — конфигурации систем
- `solutions/` — решения проблем

---

### ✅ Результаты обсуждения (2026-03-26 12:24 UTC)

**Ключевое понимание:**
> `~/projects/total-recall/docs` — это **документация плагина total-recall**, а не библиотека знаний системы!
> `~/projects/total-recall` — это **GitHub репозиторий плагина**, а не хранилище знаний!

**Цепочка рассуждений:**

1. **Осознание проблемы:**
   - total-recall/docs — документация плагина (как работает, архитектура, API)
   - total-recall — GitHub репозиторий плагина
   - Это НЕ библиотека знаний системы!

2. **Вариант A (отдельный репозиторий):** ❌
   - Создать `~/projects/system-knowledge/` или `~/projects/knowledge-base/`
   - Проблема: over-engineering, ещё один репозиторий, сложная интеграция с total-recall
   - Вердикт: Слишком сложно

3. **Вариант B (существующие файлы):** ❌
   - Расширить SYSTEM.md, MEMORY.md
   - Проблема: файлы станут огромными, смешиваем типы знаний, сложно найти
   - Вердикт: Плохая практика

4. **Вариант C (гибрид):** ⚠️
   - Создать `workspace/knowledge/` для знаний
   - Проблема: всё ещё нужно интегрировать с total-recall
   - Вердикт: Лучший из трёх, но не идеален

5. **Настоящая проблема:**
   - total-recall — это **плагин для flashback из Mem0**
   - total-recall только **запрашивает Mem0** (Neo4j + Qdrant)
   - total-recall **НЕ индексирует файлы**
   - total-recall **НЕ загружает документацию**

6. **Три решения:**
   - **Решение 1 (простое):** Хранить знания в Mem0 (с метками type: instruction/configuration)
     - Плюсы: total-recall уже умеет работать с Mem0
     - Минусы: Mem0 для уроков, не для инструкций
   
   - **Решение 2 (правильное):** Расширить total-recall для индексации файлов
     - Плюсы: чёткое разделение, единая точка доступа, масштабируемо
     - Минусы: нужно расширять total-recall, индексировать файлы, синхронизировать
   
   - **Решение 3 (радикальное):** Не хранить инструкции
     - Плюсы: простота, нет дублирования
     - Минусы: нужно искать вручную, нет автоматического flashback

7. **Предложение критика:**
   - Выбрать Решение 2 (расширить total-recall)
   - Создать `workspace/knowledge/` для знаний системы
   - Расширить total-recall для индексации знаний
   - Оставить Mem0 для уроков
   - Не создавать новый репозиторий

**Что остаётся ОТКРЫТЫМ:**
- ❓ Какое решение выбрать? (1, 2, или 3)
- ❓ Если Решение 2 — как именно расширять total-recall?
- ❓ Структура `workspace/knowledge/` — какие папки?
- ❓ Как индексировать файлы в Qdrant?
- ❓ Как синхронизировать при изменении файлов?

---

### ✅ Результаты обсуждения (2026-03-26 13:22 UTC)

**Ключевое понимание:**
> bge-reranker-v2-m3 работает через embeddings API с cosine similarity, но не как specialized reranker!

**Текущее использование bge-reranker:**
- ✅ `flashback_focus (--focus "тема")` — с reranker (Qdrant + bge-reranker-v2-m3 через embeddings + cosine similarity)
- ❌ `flashback (--category "dev")` — без reranker (только Neo4j запрос)

**Цепочка рассуждений:**

1. **Анализ вариантов (1-4):**
   - Вариант 1 (Mem0 как special Evidence) — ❌ нарушает назначение Mem0
   - Вариант 2 (TRANSFERRED.md) — ❌ бесполезно без интеграции
   - Вариант 3 (workspace/knowledge/transferred/) — ⚠️ зависит от решения вопроса 2
   - Вариант 4 (USER.md) — ❌ нарушает назначение USER.md

2. **Проблема всех вариантов:** Нарушают принцип "не дублировать функции системы"

3. **Предложение критика:** Решение 0 (не хранить отдельно) + Решение 2 (расширить Mem0)
   - Не создавать отдельную систему для "переданных знаний"
   - Использовать Mem0 с меткой `evidence_type: "transferred"`
   - Чётко разделять по меткам: empirical vs transferred vs documented

4. **Вопрос о механизме вызова:** Как будут вызываться знания с `evidence_type: "transferred"`?

5. **Сравнение Вариантов A и B:**
   - **Вариант A (отдельный flashback):** Проще, гибче, но медленнее (2 вызова)
   - **Вариант B (объединённый с приоритетами):** Быстрее, умнее, но сложнее (сортировка)

6. **Ключевое открытие:** bge-reranker не используется для category flashback!
   - Сейчас: только Neo4j запрос по категории
   - Можно: добавить reranker как в --focus

7. **Итоговое решение:** Вариант B с reranker
   - Добавить reranker для category flashback
   - Объединить category + transferred в один вызов
   - Reranker для всех результатов
   - Сортировка по relevance_score

8. **Проверка bge-reranker endpoint:**
   - ❌ Ollama не имеет native `/api/rerank` endpoint
   - ❌ Модель не работает как specialized reranker через `/api/generate`
   - ✅ **Модель работает через `/api/embeddings` с cosine similarity**
   - ✅ **Это рабочее решение для total-recall**

**Решение:**
- Использовать Mem0 с меткой `evidence_type: "transferred"` для знаний от Олега
- Использовать bge-reranker-v2-m3 через embeddings API с cosine similarity для reranking
- Объединённый flashback с приоритетами:
  - Meta > Principle > Lesson > Conclusion
  - transferred > empirical > documented
- Один вызов flashback с embeddings + cosine similarity для всех результатов

**Что нужно реализовать:**
1. Добавить `evidence_type: "transferred"` в Mem0 (уже поддерживается)
2. Обновить `store.py` — использовать embeddings + cosine similarity вместо native rerank API
3. Обновить config — использовать `/api/embeddings` endpoint
4. Объединить category + transferred в один вызов
5. Сортировать результаты по relevance_score + типу

**Статус:** 🟢 Решён (Вариант B с embeddings + cosine similarity)
